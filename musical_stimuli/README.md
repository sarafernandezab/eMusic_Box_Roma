# Musical stimuli

This folder contains the musical materials used by the eMusic Box Roma (eMB Roma), including example MIDI files, MIDI-derived text files formatted for playback by the eMB, and the Python conversion pipeline used to generate them.

The musical stimuli are designed to be synchronized with the rotational data from the eMBs. During playback, the Max/MSP patch reads the MIDI-derived text files frame by frame and maps each frame to the current rotational position of the eMB handle.

## Musical stimulus format

All musical materials must be formatted so they can be interpreted and synchronized with the rotational data from the eMB. The eMB system uses simplified text-based representations converted from standard MIDI files.

By default, one full rotation of the eMB corresponds to one musical bar divided into 16 equally spaced steps. This matches the sixteenth-note structure of a 4/4 bar. Each step represents a fixed temporal frame in which one or more musical events may occur.

Each note is represented using MIDI-style values:

```text
pitch velocity duration
```

where:

- `pitch` is the MIDI pitch value,
- `velocity` determines the loudness or intensity of the note,
- `duration` specifies the temporal length of the note.

Polyphonic events are represented by including multiple pitch-velocity-duration triplets in the same row. Frames without musical events are represented as silence, with MIDI values set to zero.

## Example stimuli

The folder `examples_MIDI/` contains example MIDI files for different instrumental parts. The examples include versions of:

- *Thriller*
- *Freed from Desire*
- *Get Down on It*
- *We Found Love*

The MIDI files are organized as separate instrument streams, such as drums, bass, keyboards, and violin or melody. Each instrumental part should be saved separately before conversion.

The corresponding eMB-ready text files are provided in:

```text
example_MIDI_derived_text_eMB/
```

These files can be loaded directly by the Max/MSP patch according to the paths specified in the global configuration file.

## Converting MIDI files to eMB text files

The conversion from MIDI to eMB-compatible text files is performed with:

```text
pipeline_convert_MIDI_to_eMBtxt.py
```

This script converts standard MIDI files into simplified text files that can be read by the Max/MSP patch during eMB playback.

The conversion pipeline supports:

- conversion of MIDI files separated by instrument,
- polyphonic MIDI sequences,
- batch conversion of multiple MIDI files,
- extraction of stimulus metadata such as tempo and number of frames,
- generation of text files compatible with the eMB rotational mapping.

If a MIDI file contains multiple instruments, the instrumental parts should be separated into different files before conversion. Filenames should indicate the corresponding song and instrument, for example:

```text
Freed_bass.mid
Freed_drums.mid
Freed_keyboard.mid
Freed_violin.mid
```

The generated text files should follow the same naming logic:

```text
Freed_bass.txt
Freed_drums.txt
Freed_keyboard.txt
Freed_violin.txt
```

## Using the stimuli in Max/MSP

At the start of an eMB session, the Max/MSP patch loads the MIDI-derived text files according to the paths defined in the global configuration file.

During playback:

1. the eMB streams rotational data to the computer,
2. the Max/MSP patch discretizes the angular position into temporal frames,
3. the corresponding row of the MIDI-derived text file is read,
4. the musical event for that frame is played through the assigned MIDI instrument stream.

This structure allows individual instrument streams to be flexibly assigned, combined, or synchronized across one or two eMB devices.

---

## Notes on editing stimuli

The MIDI-derived text format can be edited manually in any text editor or regenerated by modifying the Python conversion pipeline.

For example, to transpose a stimulus down by one octave, subtract `12` from the pitch value of each note.

When editing files manually, preserve the row structure so that the number and order of frames remain compatible with the Max/MSP patch and the expected rotational mapping.
